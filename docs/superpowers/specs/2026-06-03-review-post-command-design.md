# 블로그 포스트 리뷰 커맨드 (`/review-post`) — 설계서

- 작성일: 2026-06-03
- 작업 브랜치: `feat/review-post-command`
- 대상 산출물:
  - `.claude/review_post.py` (결정적 검사 스크립트)
  - `.claude/commands/review-post.md` (`/review-post` 슬래시 커맨드)
  - `.claude/commands/review-post-all.md` (`/review-post-all` 슬래시 커맨드)

## 1. 목적 및 스코프

블로그 포스트(`src/content/posts/*.md`)의 **내용·문체·SVG**를 점검하는 온디맨드 리뷰 도구를 만든다. 다익스트라 1·2편에서 수작업으로 했던 문체 교정(줄표 남발 제거, 깨진 굵게 수정 등)을 규칙화해 반복 가능하게 만드는 것이 핵심 동기다.

**포함:**

- 결정적 검사(기계 판정)와 LLM 비평(사람 판단)을 한 리포트로 묶는 하이브리드 리뷰
- 두 진입점: `/review-post`(main 대비 변경된 포스트), `/review-post-all`(전체 포스트)
- 심각도 표시(🔴 필수 / 🟡 권장 / 🟢 참고)와 파일·줄 위치

**제외 (의도적 YAGNI):**

- 자동 훅(PostToolUse/Stop/pre-commit) 연동 — v1은 전부 수동 실행
- 자동 수정(`--fix`) — v1은 리포트만. 수정은 사용자가 보고 판단
- 포스트 외 파일(컴포넌트, 설정 등) 리뷰
- CI/GitHub Actions 연동

## 2. 동작 방식 (하이브리드, 전부 수동)

자동 개입은 없다. 사용자가 슬래시 커맨드를 실행할 때만 동작한다.

```
/review-post            → main 대비 변경된 src/content/posts/*.md
/review-post <slug>     → 특정 글 하나 (예: /review-post dijkstra-2)
/review-post-all        → 모든 src/content/posts/*.md
```

각 커맨드의 처리 흐름:

1. 대상 `.md` 파일 목록을 결정한다.
   - `review-post`: `git diff --name-only main...HEAD` + 워킹트리 변경분에서 `src/content/posts/*.md` 추출. 인자(slug)가 있으면 그 파일로 한정.
   - `review-post-all`: `src/content/posts/*.md` 전체.
2. 대상 파일들을 인자로 `.claude/review_post.py`를 실행해 **결정적 검사 결과**(JSON 또는 텍스트)를 받는다.
3. Claude가 각 포스트 본문과 참조된 SVG를 읽어 **LLM 비평**을 더한다.
4. 둘을 합쳐 포스트별 **통합 리포트**를 출력한다.

## 3. 구조 (스크립트 + 커맨드 분리)

결정적 검사는 코드로, 판단은 LLM 프롬프트로 분리한다.

### 3.1 `.claude/review_post.py`

- 입력: 검사할 `.md` 파일 경로들(`sys.argv[1:]`). 경로가 없으면 종료 코드 0으로 조용히 종료.
- 처리: 각 파일을 읽어 아래 **결정적 검사 항목**을 수행.
- 출력: 사람이 읽기 쉬운 텍스트 리포트를 stdout으로. 각 발견 항목은 `심각도 · 파일:줄 · 메시지` 형식.
- 의존성: 표준 라이브러리만 사용(`re`, `pathlib`, `sys`, `xml.dom.minidom`, `json`). 외부 패키지 없음.
- 스타일: 기존 `.claude/inject_datetime.py`와 동일한 결(단순 스크립트, 예외는 조용히 무시하지 말고 해당 검사만 건너뛰되 사유 표시).
- 종료 코드: 발견 사항 유무와 무관하게 0(리포트 도구이므로). 단 🔴 필수 항목 존재 시 비제로 종료를 옵션(`--strict`)으로 둘 수 있음(향후 pre-commit 재활용 대비, v1에서는 미사용).

### 3.2 `.claude/commands/review-post.md` / `review-post-all.md`

- Claude Code 프로젝트 슬래시 커맨드(마크다운 프롬프트).
- 내용: 대상 선정 방법, `review_post.py` 실행 지시, SVG·본문 읽기 지시, LLM 비평 루브릭, 출력 포맷(심각도/위치).
- 두 파일은 "대상 선정" 부분만 다르고 나머지 루브릭은 공유. 중복을 줄이기 위해 공통 루브릭을 한쪽에 두고 다른 쪽에서 동일 기준을 참조하도록 기술.

## 4. 검사 항목

### 4.1 결정적 검사 (`review_post.py`)

| # | 검사 | 심각도 | 설명 |
|---|---|---|---|
| D1 | 깨진 굵게 | 🔴 | `구두점 + ** + 글자` 패턴(예: `)**글자`). 정규식 `\p{P}\*\*[\p{L}\p{N}]` 상당. CommonMark에서 굵게가 적용되지 않음 |
| D2 | 줄표(—) 남발 | 🟡 | 글당 `—`(em dash) 개수 카운트. 임계치(예: 본문 100줄당 N개) 초과 시 경고 |
| D3 | 강조 과다 | 🟢 | `**굵게**`·`*기울임*` 출현 밀도. 과도하면 참고용 경고 |
| D4 | SVG 유효성 | 🔴 | 참조된 모든 `.svg`를 `xml.dom.minidom`으로 파싱, well-formed 여부 |
| D5 | 에셋 경로 | 🔴 | 모든 `![](/images/...)`·SVG 경로가 `public/` 아래 실제 존재하는지 |
| D6 | 내부 링크·앵커 | 🟡 | `/blog/<slug>` 링크가 존재하는 포스트인지, `#앵커`가 본문 제목에서 생성되는 슬러그와 일치하는지 |
| D7 | frontmatter | 🟡 | `title·date·description·tags·category·difficulty` 누락 여부, `description` 길이(너무 짧/긺) |
| D8 | 수식 짝 | 🟡 | `$`(인라인)·`$$`(블록) 구분자 짝이 맞는지(홀수 개 등) |

비고:
- D1·D6·D8의 정확한 임계치/슬러그 규칙은 구현 단계에서 실제 포스트로 검증해 확정한다.
- 앵커 슬러그 규칙은 사이트가 사용하는 방식(한글 제목 → id 변환)을 실제 렌더 결과로 확인해 맞춘다.

### 4.2 LLM 비평 (커맨드 프롬프트 루브릭)

| # | 항목 | 설명 |
|---|---|---|
| L1 | 문체(AI 신호) | 줄표 남발, 경구식 섹션 마무리, 과한 비유/수사("형제처럼", "두 얼굴"), 대칭 문장, 과한 강조. 문제 문장을 구체적으로 인용해 지적 |
| L2 | 설명 흐름·명료성 | 논리 도약, 빠진 전제, 어려운데 안 풀어준 부분, 너무 긴 문장 |
| L3 | 용어·어체 일관성 | 용어 혼용(노드/정점, 엣지/간선 등), 어체(~다 평서체) 통일 |
| L4 | SVG ↔ 본문 일치 | SVG의 레이블·수치·캡션이 본문 예제/주장과 일치하는지(예: 그래프 가중치, 거리 값) |
| L5 | 제목·description 적합성 | 제목과 frontmatter description이 실제 내용을 잘 대표하는지 |

비고:
- L1의 "AI 신호" 기준은 다익스트라 1·2편 교정에서 도출한 패턴을 명문화한다(줄표→마침표/접속사, 경구 마무리 제거, 비유 최소, ~다 유지).
- 수학적 정확성(증명·수식 오류)은 v1에서 적극 판정하지 않는다. 명백한 모순이 보이면 🟢 참고로만 언급.

## 5. 출력 포맷

포스트별로 묶어 출력한다. 예시:

```
## 리뷰: src/content/posts/dijkstra-2.md

🔴 필수 (2)
- [D1] L73  깨진 굵게: ")**가" — 닫는 ** 앞 구두점 + 뒤 글자
- [D5] L108 에셋 없음: /images/dijkstra-2/bfs-edge-split.svg

🟡 권장 (3)
- [D2] 줄표(—) 21회 — 임계치(12) 초과. AI 문체 신호
- [L2] L40  한 문장이 너무 길어 두세 개로 나누면 읽기 쉬움
- [L3] L88  "정점"과 "노드"가 혼용됨

🟢 참고 (1)
- [L4] complexity-sparse-dense.svg 의 캡션이 본문 표현과 미세하게 다름

요약: 🔴 2 · 🟡 3 · 🟢 1
```

- 결정적(Dn)·LLM(Ln) 항목 번호를 붙여 출처를 표시.
- 변경된 포스트가 없으면 그 사실을 알리고 종료.

## 6. 컴포넌트 경계 (단위별 책임)

- `review_post.py`: 텍스트/파일 기반의 **결정적 판정**만 담당. LLM·네트워크 의존 없음. 입력=경로 목록, 출력=텍스트 리포트. 단독 실행·테스트 가능.
- `review-post*.md`: **오케스트레이션과 판단**. 대상 선정 → 스크립트 호출 → 본문/SVG 정독 → 비평 → 통합 출력. 결정적 검사 로직을 직접 구현하지 않고 스크립트에 위임.

## 7. 검증 방법

- `review_post.py`를 기존 포스트들(특히 다익스트라 1·2)에 직접 실행해, 알려진 사례가 올바르게 잡히는지 확인:
  - D1: 과거 `)**글자` 버그가 있던 커밋 상태에서 탐지되는지(현재는 수정됨 → 0건이어야 함)
  - D4/D5: 실제 SVG·이미지 경로로 통과/실패가 맞는지
- 일부러 깨뜨린 임시 마크다운(깨진 굵게, 없는 이미지, 짝 안 맞는 `$`)으로 각 검사가 발화하는지 확인.
- `/review-post`가 main 대비 변경 포스트를 올바르게 선별하는지(브랜치에서 한 글만 수정 후 실행).

## 8. 향후 확장 (v1 제외, 기록만)
